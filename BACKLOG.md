# BACKLOG

Ideas for improvements and new functionality. Not committed work — items move to SESSION_STATE.md's pending blocks once actually scoped and scheduled.

**North star:** the end result people see is a trust signal pasted into a social media post about a repo (see SPEC.md §5 "LinkedIn trust block") — something that makes someone who's never met the maintainer feel safe downloading/using the repo. Every improvement here should be evaluated against that: does it make the signal more credible, more visible, or easier to produce on demand?

## Skill improvements

Changes to what the repo-trust skill installs or does when run against a target repo.

- [ ] Branch protection automation revisited — offered once already (SPEC.md §1.6) and declined for the repo-trust repo itself (solo maintainer). Revisit if collaborators join or Scorecard's branch-protection check becomes worth chasing.
- [ ] Monthly audit as a scheduled prompt — SPEC.md §5 already defines the audit prompt text; today it's run manually. *Decision 2026-07-05: stays manual — user chose no scheduling infrastructure over a cloud agent or external-app option. Revisit if manual cadence lapses in practice.*
- [ ] CodeQL scanning — explicitly excluded from v1 scope (SPEC.md §7: "future extensions, proposed separately with evidence of need"). Would add static analysis alongside Trivy's vuln/secret/license scan. *Decision 2026-07-05 (morning): deferred — this repo is markdown-only. Reversed 2026-07-05 (evening): user manually enabled GitHub's default code-scanning setup for this repo, which surfaced that CodeQL's "actions" language target analyzes GitHub Actions workflow files — so a workflow-only repo is not actually a null target. User now wants the skill itself to install, run, and validate CodeQL (not rely on manual per-repo toggling). Scoped into blocks C1–C2.*
- [ ] Signing / attestations (e.g. Sigstore/cosign for release artifacts) — same §7 exclusion. Would strengthen the SBOM's trust chain beyond "attached to the release." *Decision 2026-07-05: brought into scope, but as GitHub-native artifact attestations, not raw cosign — see Fable-sourced entry below. Scoped into pending blocks A1–A2.*
- [ ] License policy enforcement — today Trivy's license scan is report-only. Could add an allow/deny list that gates the release like the vulnerability severity gate does. *Decision 2026-07-05: deferred to second-repo validation — this repo has no manifests, so there is nothing to gate and no way to test the gate honestly.*
- [ ] Richer SBOM for manifest-less repos — SPEC.md §3.3 accepts a thin/empty SBOM when no package manifests exist (e.g. this repo). Investigate whether CycloneDX can usefully describe non-package assets (workflows, skill files) for these cases. *Decision 2026-07-05: deferred — low north-star value (a stranger scrolling a feed doesn't gain confidence from a synthetic component list; the honest "thin SBOM, here's why" report from R1 covers it better).*
- [ ] Private-repo degraded mode UX — SPEC.md §3.1 defines the fallback (Security-tab-only, no badge) but the user-facing report of what's missing and why could be clearer. *2026-07-05: folded into block R1 — the schematic report format defines how degraded/missing signals render (⚠️ line + reason) instead of designing a separate UX.*
- [ ] Schematic, emoji-formatted reporting — today's reports (install summary, SPEC.md §5 audit/LinkedIn prompts) are plain prose ("3 líneas"). Replace with a consistent structured format: status glyphs (✅ pass / ⚠️ degraded / ❌ fail) per check, grouped under short headers, so a report is scannable in a glance rather than read start to finish. *Scoped 2026-07-05 into pending blocks R1/R3.*
- [ ] Skill runs + reports + advises, not just installs — SPEC.md §7 scopes the skill to installation; the audit prompt (§5) exists but is a separate manual step bolted on afterward. Fold "run the checks, report live results, propose concrete next improvements" into the core flow so one invocation both installs what's missing *and* operates: surface current findings (Trivy severity counts, Scorecard per-check breakdown, SBOM/release freshness) and recommend specific next actions (e.g. "Scorecard is 5.8 — enabling branch protection would raise the code-review check" or "2 HIGH findings in workflow X — here's the fix"), rather than leaving that analysis to a separately-invoked audit. *Scoped 2026-07-05 into pending blocks R2/R3.*

### Fable-sourced candidates (added 2026-07-05)

Proposed by the planning model against the north star; each entry says what it adds to the pasted signal.

- [ ] GitHub artifact attestations (SLSA build provenance) — concretizes the "Signing / attestations" item above: `actions/attest-build-provenance` on the release job attests the SBOM/release assets, free on public repos, no key management. North star: turns "trust me" into a one-command check a stranger can run (`gh attestation verify`), and that command itself is pasteable into a post. *Scoped into blocks A1–A2.*
- [ ] "Verify it yourself" section — a short README block plus a line in the trust block with copy-paste verification commands (Scorecard API curl, SBOM download link, `gh attestation verify`). North star: a claimed signal is noise; a checkable one is credible. *Scoped into blocks A1–A2 (spec + implementation travel together).*
- [ ] Scorecard score history + delta reporting — append the live score to `scorecard_history.jsonl` on every audit; audit report and trust block show "5.8 → X desde <fecha>". North star: SPEC.md §3.2 already says "the score's upward trend is the content, not a perfect 10" — this makes the trend citable instead of anecdotal. *Scoped into blocks D1–D2.*
- [ ] Dependency-review action on PRs — `actions/dependency-review-action` blocks known-vulnerable dependency changes at PR time; adds a "gated at PR time" line to the signal. Near no-op on manifest-less repos like this one — deferred to second-repo validation, alongside CodeQL and license policy.
- [ ] Split CodeQL's `analyze` step into `upload: false` + `actions/upload-artifact` — discovered installing on `initium` (2026-07-06, spec v2.3): on a private repo without a Code Security license, `continue-on-error: true` can't distinguish "no license" from "CodeQL actually found something", since one action call both computes and uploads. Mirroring `scorecard.yml`'s own `results_file`/`upload-artifact` pattern would preserve real findings as a downloadable artifact even without Security-tab access. Not solved in v2.3 for time; codeql.yml currently just gets skipped entirely on unlicensed private repos instead (§0.2.8), losing CodeQL's value there rather than degrading gracefully.
- [ ] CodeQL Action v3 → v4 — GitHub's own deprecation notice (surfaced in C2's first run, 2026-07-05): v3 deprecated December 2026. Not urgent (12+ month runway), but re-resolve and bump at the next audit touching codeql.yml.
- [ ] Pin `actions/attest` directly instead of the `attest-build-provenance` wrapper — discovered at A2 implementation (2026-07-05): v4 of attest-build-provenance is a thin wrapper over actions/attest; pinning the underlying action is one less indirection in the supply chain. Spec decision, low urgency.
- [ ] OpenSSF Best Practices badge (bestpractices.dev) — a second third-party citable number next to Scorecard. Weaker credibility (self-assessment questionnaire, manual web signup); the skill could at most pre-fill evidence for the answers. Parked unless a post needs a second badge — recommend not pursuing now.

## Project roadmap

Work on the repo-trust repo/skill as a project, beyond what any single install does.

- [ ] Second-repo validation — SPEC.md §8 gates "packaging as a skill" on 2 repos validated. *2026-07-05: second repo confirmed as `initium` (private, no package manifests, real Python source in scripts/) — scoped into blocks I1-I5. Chosen despite being private/manifest-less because it validates two things repo-trust couldn't validate on itself: the degraded-mode (private repo) path, and CodeQL detection when a real language exists without a manifest.*
- [ ] Third validation case: public repo with real package manifests — still needed to validate SBOM richness (non-zero components), a real license scan, and CodeQL with a manifest-mapped language (not just source-file detection). Neither repo-trust nor initium can prove this path. Not yet scoped; needs the user to name a candidate repo.
- [ ] Automated tests for the skill logic — SPEC.md §4's acceptance tests are currently run manually per install; consider a harness that exercises them against a scratch repo.
- [ ] Usage walkthrough / example — a worked example (screenshots or transcript) of a fresh install for someone evaluating the skill before installing it.
- [ ] Contribution guidelines — none yet; needed if this moves from personal skill to something others install or PR against.
